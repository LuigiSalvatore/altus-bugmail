# System Overview

- **Purpose**: Lightweight Bugzilla web tracker providing a Flask‑based UI and JSON API to browse, update and manage bugs.
- **Runtime Architecture**:
  - Single‑process Python application.
  - Uses a background auto‑refresh thread to periodically sync with the remote Bugzilla server.
  - In‑memory state (`_state`) holds configuration, cached bug data and a threading lock.
  - A single‑instance guard socket ensures only one server runs per user.
- **Major Subsystems**:
  - **Web UI** – static files served from `webapp/`.
  - **API Layer** – Flask routes under `/api/*`.
  - **Bugzilla Client** – `apiBugzilla.py` wrapper around Bugzilla REST API.
  - **Persistence** – JSON files `bugzilla_config.json` and `bugzilla_data.json`.
  - **Auto‑Refresh** – daemon thread updating cached data every minute.
- **Execution Model**:
  - Server start → acquire guard socket → launch background thread → start Flask HTTP server.
  - All mutable state accessed under `_state['lock']`.
- **Threading Model**:
  - Main Flask thread handles HTTP requests.
  - One daemon thread (`auto_refresh_loop`) updates data.
- **Hardware / OS Interaction**: None beyond standard file I/O and socket binding (Windows).
